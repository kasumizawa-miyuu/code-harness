# REFLECTION.md — 项目反思报告

## 一、Superpowers 技能评估

### 发挥最大作用的技能

**1. brainstorming（头脑风暴）**

这是整个项目中价值最高的技能。它没有让我直接跳进实现，而是强制性地追问了三个关键问题："你究竟想做什么？"、"选哪个方向？"、"为什么选这个？"。特别是它对"移除 LLM 后还能测试吗"这一判据的反复强调，从根本上改变了架构设计方向——从"写一个能用的 agent"变成了"写一个可测试的 harness 内核"。如果没有这个追问，我大概率会写出一个依赖 LLM 作判断的"提示词包装器"，而不是一个真正的工程系统。

**2. test-driven-development（测试驱动开发）**

TDD 在 AI 协作场景下意外地成了一种"质量控制协议"。每个 task 的流程是：先写失败测试 → 运行确认红色 → 让 subagent 实现 → 运行确认绿色。这个流程的强制力在于：它让"通过"的定义变得客观和可重复。在传统开发中，TDD 经常因"赶进度"而被跳过；但在 AI 协作中，TDD 反而更容易执行——因为写测试和写实现可以由不同实体完成，且测试失败时 AI 不会像人类一样找借口。

**3. writing-plans（编写计划）**

将整个项目分解为 17 个 2-5 分钟可完成的 task，每个 task 包含明确的文件路径、接口定义、验证步骤。这种颗粒度刚好匹配 subagent 的"单次会话能力边界"——太粗的 task 会让 subagent 偏离轨道，太细的 task 会引入过多的上下文切换开销。17 个 task 中，有 16 个由 subagent 一次性完成，1 个（最后的 review fix）需要人工介入。

### 形式大于实质的技能

**using-git-worktrees**：worktree 的隔离本身是有价值的——每个 task 在独立分支上开发，避免相互干扰。但本项目中 worktree 的管理开销（创建、切换、合并、清理）与收益基本持平，没有显著提升效率。对于 solo 开发者来说，普通分支 + 临时 stash 的效果相差不大。worktree 的真正价值可能体现在团队协作场景中。

## 二、TDD 在 AI 协作下的表现

**是阻碍还是放大器？** 我认为是放大器。

阻碍的一面：TDD 要求先写测试，这意味着在实现之前就要对接口有清晰的定义。这对于"探索性"或"原型"开发阶段是不自然的——有时你需要在写测试之前先看看实现能不能跑通。

放大器的一面：TDD 提供了一个"客观的完成标准"，这对 AI 协作至关重要。人类与 AI 协作的最大问题是"你以为它懂了，其实它没懂"。当 subagent 说"完成了"时，你无法判断它是真的完成了，还是只是输出了一段看起来像完成的代码。TDD 的红色/绿色信号给出了一个无可争议的客观判定：测试通过 = 完成，测试失败 = 未完成。

具体数据：本项目 44 个测试用例，全部使用 mock LLM，不依赖网络和真实 API。这意味着任何时候 `npm test` 都能给出确定性的通过/失败结果。这在传统开发中可能不算什么，但在 AI 协作场景中，它是"信任"的基础。

## 三、Subagent 工作流体验

**subagent 能自主运行多久而不偏离？**

平均每个 task 约 2-5 分钟，最长的（AgentLoop 集成）约 10 分钟。超出这个时间窗口，subagent 开始出现"忘记上下文"、"重复已有代码"、"偏离 spec"等问题。

**什么导致了偏离？**

- **上下文窗口限制**：subagent 的上下文在每次调用后重置，跨 task 的共享知识（如架构约定、命名规范）需要人工在 prompt 中重复传递。
- **任务边界模糊**：当 task 涉及多个文件时，subagent 容易只修改一个文件而遗漏其他文件。
- **"幻觉完成"**：subagent 偶尔会声称"已完成"，但实际代码中缺少关键逻辑——这通常发生在它认为某个步骤"太简单"而不需要实现时。

**什么颗粒度最优？**

单个文件 + 单个接口 + 2-5 分钟实现时间。这个颗粒度刚好在 subagent 的"注意力跨度"内，也正好对应一次 TDD 循环（红-绿-重构）。

## 四、SPEC / PLAN 质量对实现质量的影响

**正面案例：** Config 模块的 SPEC 中明确定义了"环境变量覆盖优先级高于配置文件，默认值最低"。这个清晰的定义让 subagent 一次性实现了正确的逻辑，无需返工。

**反面案例：** AgentLoop 的 SPEC 中关于"自适应重试"的条件写的是"同一错误重复出现时提前停机"，但没有明确定义"同一错误"的判定标准。subagent 首次实现用了精确字符串匹配（`===`），导致测试中出现"失败信息只有微小差异但实际是不同错误"时无法正确重试。这个缺陷在 code review 阶段被发现，修正为支持更宽松的匹配策略。

**教训：** SPEC 中"模棱两可"的描述会直接导致 subagent 做出"看似合理但实际错误"的实现。含糊 = bug。

## 五、Prompt / Context 策略

**最有效的策略：** 在 subagent 的 prompt 中明确包含"输出格式"约束。例如：

```
Output ONE action per response. After receiving feedback, adjust your approach.
```

这个约束直接决定了 ActionParser 能否正确解析 subagent 的输出。没有格式约束时，subagent 会输出花哨的 Markdown 格式文本，导致解析失败。

**为什么有效：** subagent 的工作方式是"生成下一个 token"，它天然倾向于输出"看起来像回事"的内容。明确的格式约束相当于一个"输出过滤器"，把它的创造力限制在可解析的范围内。

## 六、凭据与分发

**这两个要求迫使我想清楚了哪些问题？**

凭据管理迫使我想清楚了一个原本会忽略的问题：**API Key 的完整生命周期**。录入（隐藏输入）→ 存储（钥匙串）→ 读取（加载到内存）→ 使用（HTTP 请求头）→ 清除（删除）。每个环节都有自己的威胁模型，缺一个环节就不安全。

分发迫使我想清楚了另一个问题：**别人拿到我的代码后，如何从零开始运行？** 这不仅仅是写一个 Dockerfile 或 npm 包的问题，而是"如何让 key 在目标机器上安全地配置"的问题。Docker 场景下，key 不能内嵌在镜像中；npm 场景下，keytar 的原生模块依赖可能导致安装失败——这些都是在分发设计中必须考虑的工程细节。

## 七、如果重做，我会改变什么？

1. **worktree 分支应该在开发完成后立即推送到 GitHub 并创建 PR，再合并到 main。** 本项目中分支先合并后推送，导致无法创建 PR 记录工作流。这是一个应该避免的流程错误。

2. **在 SPEC 中增加"接口契约"章节。** 当前 SPEC 的功能规约是描述性的（"应该做什么"），不是契约性的（"输入什么 → 输出什么"）。如果加入 TypeScript 接口签名作为规范的一部分，subagent 的首次实现准确率会更高。

3. **冷启动验证应该在实现开始前完成。** 课程要求用不同 agent 验证 SPEC + PLAN 的清晰度，这个验证应该在 task 1 之前做，而不是在全部实现之后。提前做可以更早发现 spec 缺陷。

## 八、对 Superpowers 方法论的批判

**Superpowers 的核心假设：** 一个结构化的流程（brainstorming → planning → TDD → review → finish）可以驾驭 AI 的不确定性，使其产出可预测的工程结果。

**这个假设在我的项目中成立吗？** 大部分成立。TDD 和 review 的"客观完成标准"确实有效。但有一个假设在我的项目中不成立：**"subagent 可以在有限上下文内独立完成任务"**。实际上，subagent 的"独立"是有条件的——它需要非常精确的上下文（接口定义、文件位置、已有代码风格）。这些上下文如果不在 prompt 中显式传递，subagent 就会"跑偏"。

**Superpowers 的另一个盲点：** 它假设"流程"可以替代"判断"。但本项目中价值最高的 AI 交互不是编码阶段，而是 brainstorming 阶段的追问——那些"为什么"的问题。流程可以保证你"做对了"，但不能保证你"做了对的事"。后一个问题，仍然需要人类工程师的判断力。

**总结：** Superpowers 是一个优秀的"流程脚手架"，它帮我守住了 TDD、评审、计划这些在 AI 协作中容易松懈的纪律。但它不能替代我对"做什么"和"做对了吗"这两个问题的回答。这正是课程的核心命题——**当 LLM 能完成大部分编码工作时，一个工程师的真正价值在于判断力**。
